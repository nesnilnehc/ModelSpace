# ModelSpace 后续迭代方向（Brainstorm 设计草案）

**Date:** 2026-03-05  
**Status:** Approved  
**Approved by:** User  
**Method:** brainstorm-design  
**Scope:** 仅设计，不包含实现代码

## 目标

在保持当前单页前端架构（无后端）的前提下，明确 ModelSpace 下一阶段的可执行迭代方向，平衡三件事：

1. 学习与认知体验（用户能快速找到并理解模型）
2. 可维护性（避免 `src/app.js` 持续膨胀）
3. 传播与复用（让模型空间可被分享、嵌入、复用）

## 当前状态（基线）

- 已有能力
  - 3D 场景浏览（Orbit 控制、坐标轴、网格、节点、同类连线、最近邻）
  - 模型/空间多选筛选、概览模式、聚焦阅读、双语切换
  - 详情面板分层信息架构（速览/定义/判定/资料/标签）
  - 数据校验脚本 + CI（`scripts/validate-model-data.mjs` + GitHub Actions）
  - 推广图导出（含裁剪与导出视角）
- 主要约束
  - 核心逻辑集中在 `src/app.js`（约 1872 行，70+ 函数，60+ DOM 查询）
  - 自动化校验偏“结构正确”，缺少“行为正确”回归（交互/渲染层）
  - 模型数据规模已接近 100，后续继续增长时可读性与性能压力会放大
  - 当前是静态数据发布模式，增量维护与版本演进成本偏高

## 迭代方向备选

### 方案 A：内容与学习体验优先（产品向，推荐）

核心思路：
- 继续强化“从概览到学习”的路径，把 ModelSpace 从“可视化”推进到“可学习的模型导航器”。

主要动作：
- 强化空间单元的学习入口（按单元聚焦后的“推荐起手模型 + 学习路径”）
- 增加跨模型关系视图（同层/跨层、前置/后继、互补/替代）
- 提供“学习任务”模板（例如：战略决策入门路线）

优点：
- 直接提升用户感知价值，适合对外展示和知识产品化
- 与现有“概览模式 + 详情面板”演进方向一致（DRY）

缺点：
- 若不先治理工程结构，功能持续叠加会放大维护成本

适用场景：
- 近期目标是提升项目影响力、教育价值和内容资产沉淀

---

### 方案 B：工程可维护性优先（工程向）

核心思路：
- 先解决“单文件承载过多职责”，为后续功能迭代打底。

主要动作：
- 将 `src/app.js` 按职责拆分：状态管理、渲染管线、交互控制、导出能力
- 建立最小端到端回归（Playwright 场景：筛选、切换语言、聚焦、导出）
- 将视觉参数和交互阈值配置化，减少“硬编码散落”

优点：
- 降低后续改动风险，提升迭代速度和可预测性
- 对多人协作、PR 审查更友好

缺点：
- 用户短期感知弱，产出偏“基础设施”

适用场景：
- 近期将进入高频迭代，且希望避免质量回退

---

### 方案 C：增长与分发优先（传播向）

核心思路：
- 把“浏览器内体验”扩展为“可传播资产”，提高外部触达和复用率。

主要动作：
- 增加分享能力（带筛选条件和视角参数的 URL）
- 增加轻量嵌入模式（iframe 只读、简化控件）
- 打通发布素材链路（推广图、局部截图、主题海报模板）

优点：
- 有助于扩大曝光与引用场景（文档、博客、教学）

缺点：
- 不直接解决核心代码复杂度，长期仍需回到工程治理

适用场景：
- 近期目标是外部传播和品牌呈现

## 推荐方案与取舍

推荐采用 **A + B 的分阶段组合**：

- Phase 1（2-3 周）：先做 B 的最小闭环（模块拆分 + 回归测试骨架）
- Phase 2（2-4 周）：在新结构上落地 A 的高价值学习体验
- Phase 3（可选）：补充 C 的分发能力

原因：
- 纯 A：短期亮眼，但技术债会快速放大
- 纯 B：质量提升明显，但用户价值释放慢
- A+B：能兼顾“可持续开发”与“用户可见价值”

## 目标架构（迭代后）

在不引入后端的前提下，前端内部形成以下边界：

- `core/state`：筛选条件、选中节点、视角、语言等状态
- `core/scene`：Three.js 对象生命周期与渲染更新
- `core/interaction`：pointer/click/keyboard/fullscreen 等事件编排
- `core/export`：导出裁剪、标签缩放、图片产出
- `domain/model-data`：模型数据解析、准入信息映射、校验结果聚合
- `ui/panels`：筛选面板、详情面板、校验面板、view dock

## 里程碑（建议）

### M1：工程基线稳固（优先）

- 拆分 `src/app.js` 为多个职责模块（不改变外部行为）
- 增加 3-5 条关键交互回归（语言切换、筛选、概览聚焦、导出入口）
- 输出模块边界文档与事件流图

验收：
- 现有功能行为不回退
- 关键交互可自动化验证
- 代码审查可按模块进行

## 本轮实施范围（2026-03-05）

为控制改动风险，本轮只做 M1 的最小闭环：

1. 抽离导出能力为独立模块（先从 `app.js` 提取最稳定的一块能力）
2. 增加最小 E2E 冒烟脚本（页面加载、语言切换、筛选、导出 hook）
3. 接入 `npm` 命令，作为本地回归入口

不在本轮实现：
- UI 大改
- 数据模型重构
- （初始范围）URL 分享与嵌入模式

### M2：学习体验增强

- 单元聚焦后提供“推荐学习路径”（起手模型 + 近邻模型）
- 详情面板增加“相关模型”分组（同类/邻近/跨层）
- 改善信息层级与默认展开策略，降低首屏认知负荷

验收：
- 新用户在 3 分钟内完成一次“选单元 -> 看模型 -> 扩展学习”
- 详情面板滚动与操作成本下降（定性 + 行为数据）

### M3：分发能力（可选）

- 支持 URL 参数完整恢复视角、筛选、语言状态
- 输出“嵌入模式”页面（简化控件，保留阅读核心）
- 扩展导出能力（单元/分类定向导图）

验收：
- 分享链接跨设备可复现同一视图
- 嵌入页面可稳定加载且交互可控

## 风险与缓解

- 风险：模块拆分引入行为回退  
  缓解：先建回归脚本，再分阶段拆分（每阶段可回滚）

- 风险：数据规模扩大导致渲染负担上升  
  缓解：增加可见节点上限策略与分层渲染开关

- 风险：功能扩展挤压文档维护  
  缓解：每个里程碑同步更新 `docs/architecture` 与 `project-file-map`

## YAGNI 与 DRY 约束

- YAGNI：本轮不引入后端、不做多用户协作、不做复杂图算法。
- DRY：复用已有 `app3d/*` 模块与现有交互模型，优先“拆分重组”而非“重写”。
