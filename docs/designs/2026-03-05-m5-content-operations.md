# ModelSpace M5：内容治理与数据运营（设计草案）

**Date:** 2026-03-05  
**Status:** Proposed  
**Approved Direction:** 路线 1（内容运营优先）  
**Method:** brainstorm-design  
**Scope:** 仅设计，不包含实现代码

## 目标

在保持静态前端架构（无后端）的前提下，把模型库从“可展示”推进到“可持续维护”：

1. 数据变更可追踪（版本化 + 变更记录）
2. 模型准入可复核（流程化 + 可审计）
3. 质量门槛可自动验证（从结构正确扩展到内容一致性）

## 当前问题（基线）

- 已有 `scripts/validate-model-data.mjs` 主要覆盖结构层校验（类别、重复名、坐标覆盖范围）。
- 模型新增/修改流程仍偏手工，缺少统一准入入口与决策记录。
- `data/model-library.js` 变更缺少标准化变更日志，历史可追溯性弱。
- 质量口径分散在文档与代码间，缺少单一“内容契约”。

## 备选方案

### 方案 A：轻量治理（文档优先）

做法：
- 只补文档模板与 PR checklist，保留现有数据文件结构。

优点：
- 成本最低，推进快。

缺点：
- 仍依赖人工执行，一致性和可审计性不足。

适用：
- 短期只想“先规范起来”，不追求自动化门禁。

### 方案 B：契约驱动治理（推荐）

做法：
- 定义统一数据契约（字段、枚举、证据、命名规范）。
- 增加“内容级”自动校验（证据条数、链接格式、引用完整度、分类一致性）。
- 追加模型变更日志生成与发布摘要。

优点：
- 在不引入后端的情况下，获得可持续的数据治理能力。
- 对未来模型扩容（100 -> 300+）更稳。

缺点：
- 首轮需要补齐脚本与文档，投入中等。

适用：
- 近期要持续新增模型，且希望每次变更都可追踪、可回滚、可审计。

### 方案 C：外部 CMS/后端治理

做法：
- 把模型数据迁移到后端或 CMS，前端只消费 API。

优点：
- 长期灵活性高，易支持多人协作。

缺点：
- 成本高，超出当前项目阶段；会显著增加运维复杂度。

适用：
- 已进入团队协作运营阶段并接受基础设施升级。

## 推荐与取舍

推荐采用 **方案 B（契约驱动治理）**，并按两周最小闭环落地。

原因：
- 比方案 A 更可持续，避免“文档有、门禁无”的失效治理。
- 比方案 C 更贴合当前“静态站点 + 轻运维”现实约束。

## 设计范围

### In Scope（M5）

- 数据契约文档化：字段字典、枚举约束、命名规则、必填项。
- 准入流程模板化：候选模型评审卡、通过/观察/拒绝结论记录。
- 校验能力扩展：内容一致性与引用完整性自动检查。
- 变更日志机制：按模型粒度记录新增/修改/删除与原因。
- CI 门禁升级：`validate + smoke + perf + content-governance-check`。

### Out of Scope（YAGNI）

- 不引入数据库、后端 API、CMS。
- 不做在线编辑后台。
- 不做自动抓取外部来源并自动入库。

## 目标产物（交付物）

1. `docs/requirements-planning/model-data-contract-v1.md`  
   内容契约：字段定义、校验规则、示例。

2. `docs/requirements-planning/model-admission-template.md`

   新增内容级校验脚本（与结构校验互补）。

5. `.github/workflows/ci.yml`（增量）  
   新增内容治理校验步骤。

## 两周里程碑（建议）

### Week 1：契约与门禁基线

- 完成数据契约文档初版（字段、枚举、示例）。
- 完成准入模板（用于新增模型评审）。
- 完成 `validate-model-content` 最小能力：
  - 证据包 key 是否存在且可映射
  - 参考资源结构完整性（作者/来源链接）
  - 中英文描述最低质量门槛（非空、长度阈值）
- 本地命令接入：`npm run validate:content`。

验收标准：
- 新脚本可在本地稳定执行。
- 对一条故意构造的坏数据能准确报错。
- 现有主干数据可通过校验。

### Week 2：变更可追踪

- 建立模型库变更日志模板与维护规则。
- 增加脚本辅助生成变更摘要（从 git diff 提取模型级变化）。
- CI 集成内容校验并在失败时阻断合并。

验收标准：
- 任意一次模型变更都能产出一条可读变更记录。
- PR 中可见“变更摘要 + 校验结果”。
- CI 对内容违规可阻断。

## 质量指标（M5 完成后）

- 新增模型准入记录覆盖率：`100%`
- 变更日志覆盖率：`100%`（新增/修改/删除均有记录）
- 内容校验失败逃逸率：`0`（主分支）
- 新模型入库审查时间：`< 15 分钟/条`（模板化后）

## 风险与缓解

- 风险：规则过严导致维护成本上升  
  缓解：先上线“警告级规则”，一周后再提升为阻断规则。

- 风险：历史数据一次性补齐成本高  
  缓解：新增数据强制、历史数据分批清理（滚动治理）。

- 风险：脚本误报影响开发效率  
  缓解：输出明确错误定位（模型名 + 字段名 + 规则 ID）。

## 首批开工清单（下一步）

1. 起草 `model-data-contract-v1.md`（先覆盖当前已使用字段，避免过度设计）。
2. 起草 `model-admission-template.md`（复用现有准入标准文档）。

4. 在 `package.json` 增加 `validate:content` 命令。
5. 将新命令接入 CI（先并行执行，再按稳定性切阻断）。
